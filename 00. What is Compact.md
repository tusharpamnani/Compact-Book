# What is Compact?

Compact is Midnight's smart contract language. It is strongly typed, statically typed, and bounded — meaning every type has a size fixed at compile time, loops must have known bounds, and recursion is not allowed. These constraints exist because every Compact program must ultimately compile into a finite zero-knowledge circuit.

> Source: [docs.midnight.network/compact/reference/compact-reference](https://docs.midnight.network/compact/reference/compact-reference)

---

## Where Compact Fits in the Midnight Stack

A Midnight smart contract has three parts:

1. **Public ledger component** — replicated on-chain state, visible to all
2. **Zero-knowledge circuit component** — proves that the ledger state resulted from a valid execution, without revealing private inputs
3. **Off-chain component** — arbitrary TypeScript code that drives the contract

Compact handles parts 1 and 2. You write Compact; the compiler generates the ZK circuits and the TypeScript bindings.

---

## What Compact Looks Like

Compact's syntax is deliberately close to TypeScript. If you know TypeScript, the surface-level grammar will feel familiar. But Compact deviates from TypeScript in important ways:

- **No bypassing the type system.** There are no missing type declarations, no `any`, no unsafe casts. The compiler enforces types at compile time, and the generated runtime enforces them again at runtime.
- **Modules are static, not dynamic.** Compact modules can have compile-time generic parameters (types and natural-number sizes). This is how the same code can be reused across different sizes and types.
- **The language is bounded.** All types have fixed sizes. All loops have bounds known at compile time. Recursion is disallowed. This is a hard requirement of ZK circuit compilation.
- **Numeric types are unsigned integers only.** Either bounded (`Uint<0..n>`) or sized (`Uint<n>`) or field-sized (`Field`).
- **Private data must be explicitly disclosed.** If a value could contain private data, it cannot flow into the public ledger without being wrapped in `disclose()`. The compiler enforces this.

---

## What the Compiler Produces

When you compile a Compact program, you get:

- A **JavaScript file** — the compiled contract logic
- A **TypeScript definition file** — so the contract can be used directly from TypeScript without an extra compilation step
- **zkir files** — zero-knowledge intermediate representation for every circuit that touches the public ledger
- **Proving keys** — one per circuit, produced from the zkir
- A **contract info JSON file** — version numbers, circuit types, and metadata

---

## The Five Kinds of Program Elements

A Compact contract can contain:

| Element | Purpose |
|---------|---------|
| `ledger` declarations | Declare what data lives on the public ledger |
| `witness` declarations | Declare callback functions provided by the TypeScript driver (for accessing private state) |
| `circuit` definitions | The operational core — functions that compile to ZK circuits |
| `constructor` definition | Runs once when the contract is deployed; initializes state |
| Module/import/type forms | Namespace management and type definitions |

---

## Key Constraint: Bounded Computation

Compact is intentionally not Turing-complete. Every program must produce a finite circuit. This means:

- No unbounded loops
- No recursion (direct or indirect)
- All type sizes fixed at compile time
- Loop bounds must be constants or sizes of constant-size objects

This is not a limitation to work around — it is the design. ZK proofs require finite circuits.
