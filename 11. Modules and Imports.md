# Modules and Imports

Compact uses a static module system for namespace management and splitting programs across multiple files. Modules are defined, exported from, and imported into programs at compile time — there is no dynamic module loading.

> Source: [docs.midnight.network/compact/reference/compact-reference#modules-exports-and-imports](https://docs.midnight.network/compact/reference/compact-reference#modules-exports-and-imports)

---

## Defining a Module

A module is a named block of program elements with its own scope:

```compact
module Math {
  export circuit add(a: Field, b: Field): Field {
    return a + b;
  }

  export circuit mul(a: Field, b: Field): Field {
    return a * b;
  }

  circuit helper(x: Field): Field {  // not exported — internal only
    return x + 1;
  }
}
```

Bindings inside a module are invisible outside it unless explicitly exported.

---

## Exporting from a Module

Two ways to export:

**1. Prefix the definition with `export`:**

```compact
module M {
  export struct Point { x: Field, y: Field }
  export circuit distance(p: Point): Field { ... }
}
```

**2. Use a separate `export` declaration:**

```compact
module M {
  struct Point { x: Field, y: Field }
  circuit distance(p: Point): Field { ... }
  export { Point, distance };
}
```

Both are equivalent. Exporting has no effect unless the module is also imported.

---

## Importing a Module

### Import everything

```compact
import Math;
// add, mul are now in scope
```

### Import specific names

```compact
import { add } from Math;
// only add is in scope, mul is not
```

### Import with a prefix

```compact
import Math prefix Math$;
// Math$add and Math$mul are in scope
// prevents name collisions
```

### Import with renaming

```compact
import { add as plus } from Math;
// plus is in scope as an alias for add
```

## Generic Modules

Modules can have generic type and size parameters, making the same module reusable across different types:

```compact
module Container<T, #N> {
  export circuit first(v: Vector<N, T>): T {
    return v[0];
  }

  export circuit last(v: Vector<N, T>): T {
    return v[N - 1];
  }
}

import Container<Field, 4>;
// first and last are now in scope, specialized to Vector<4, Field>
```

Generic modules must be specialized at the point of import. The generic parameters are visible throughout the module body.

## Importing from Files

When the import name hasn't been defined in the current file, the compiler looks for it on disk.

### By module name

```compact
import MyModule;
```

The compiler looks for `MyModule.compact` in the same directory as the importing file, then in the Compact search path.

### By file path

```compact
import "utils/Math";
// looks for utils/Math.compact relative to the importing file
```

Importing by path allows importing two modules with the same name from different locations:

```compact
import "v1/Tokens" prefix V1$;
import "v2/Tokens" prefix V2$;
// V1$transfer and V2$transfer are both in scope
```

**Rules:**
- The file must contain exactly one module definition matching the import name and nothing else (besides comments and whitespace)
- If the file is not found, it is a static error
- The Compact search path is set via `--compact-path` flag or `COMPACT_PATH` environment variable

## Include Files

`include` splices the contents of another file directly into the current file at the point of inclusion — it is not a module import:

```compact
include "shared/types.compact";
```

The included file must contain syntactically valid program elements. They are treated as if they appeared inline at the `include` point. This is useful for sharing type definitions or common declarations across contracts.

---

## Top-Level Exports

Circuits, types, and ledger fields can be exported from the contract's top level (not inside a module) to make them visible to the TypeScript driver:

```compact
export circuit transfer(to: Bytes<32>, amount: Uint<64>): [] { ... }
export struct TokenInfo { name: Bytes<32>, supply: Uint<64> }
export ledger totalSupply: Uint<64>;
```

**Rules for top-level exports:**
- No two exported circuits at the top level can share the same name — it is a static error
- Generic circuits cannot be exported from the top level — they must be specialized first
- Size parameters (`#N`) are dropped when exporting generic structs to TypeScript — only type parameters carry through

---

## The Standard Library

The standard library is a special built-in module, not a file:

```compact
import CompactStandardLibrary;
```

It does not exist on disk and does not need to be on the Compact search path. The compiler knows where to find it.

---

## Module Order

The order of program elements within a file is mostly unimportant — circuits can call other circuits defined later in the file. However:

- A module must be **defined before** any import of that module
- Program-defined types used as generic arguments in an import must be **defined before** the import form

## Practical Example — Splitting a Contract Across Files

**`state.compact`**

```compact
module State {
  enum STATUS { UNSET, SET }
  ledger status: STATUS;
  ledger value: Field;

  export circuit init(v: Field): [] {
    value = disclose(v);
    status = STATUS.SET;
  }

  export circuit get(): Field {
    assert(status == STATUS.SET, "Not initialized");
    return value;
  }
}
```

**`main.compact`**

```compact
pragma language_version 0.22;
import CompactStandardLibrary;
import State;

constructor(v: Field) {
  init(v);
}

export circuit getValue(): Field {
  return get();
}
```

## Quick Reference

| Syntax | Effect |
|--------|--------|
| `module M { ... }` | Define module `M` |
| `export circuit f(...) { ... }` | Export `f` from enclosing module/top-level |
| `export { f, g }` | Export `f` and `g` from enclosing module |
| `import M` | Import all exports of `M` into scope |
| `import { f } from M` | Import only `f` from `M` |
| `import M prefix P$` | Import all of `M` with prefix `P$` |
| `import { f as g } from M` | Import `f` renamed as `g` |
| `import M<T, 4>` | Import generic module specialized with `T` and `4` |
| `import "path/M"` | Import `M` from a specific file path |
| `include "file.compact"` | Splice file contents inline |
| `import CompactStandardLibrary` | Import the built-in standard library |
