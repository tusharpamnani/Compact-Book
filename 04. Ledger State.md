# Ledger State

The ledger is the on-chain, public state of a Compact contract. Understanding what goes in the ledger — and what doesn't — is one of the most important design decisions you'll make when building on Midnight.

> Sources: [docs.midnight.network/compact/reference/ledger-adt](https://docs.midnight.network/compact/reference/ledger-adt), [docs.midnight.network/compact/reference/compact-reference](https://docs.midnight.network/compact/reference/compact-reference#declaring-and-maintaining-public-state), [dev.to — You're Probably Using export ledger Wrong](https://dev.to/midnight-aliit/youre-probably-using-export-ledger-wrong-4j1c)

---

## The Two Worlds

Every piece of state in a Midnight contract lives in one of two worlds:

| | `export ledger` | Private state |
|--|-----------------|---------------|
| Where it lives | Every node on the network | User's local storage |
| Who can read it | Anyone | Only the owner |
| On-chain representation | Plaintext value | Cryptographic commitment |
| How it's updated | Direct ledger assignment | Via ZK proof |
| Solidity equivalent | `public` storage variable | No equivalent |

**Privacy is the default in Compact. Disclosure is an explicit exception you must declare.** The compiler enforces this — it will reject your program if witness data flows into public ledger state without a `disclose()` wrapper.

---

## Declaring Ledger Fields

Each field in a contract's on-chain state is declared with a `ledger` keyword:

```compact
ledger val: Field;
export ledger cnt: Counter;
sealed ledger config: Uint<32>;
export sealed ledger mapping: Map<Boolean, Field>;
```

- `ledger` — declares a field in the public on-chain state
- `export` — makes the field readable from the TypeScript side of your DApp
- `sealed` — the field can only be written during contract initialization (by the constructor). Any attempt to update it from an exported circuit is a static error.

All ledger fields are automatically initialized to the default value of their type. The constructor, if any, runs after defaults are set and can override them.

## `disclose()`: The Boundary Marker

`disclose()` is not an encryption function. It is a compile-time annotation that tells the compiler: *"I know this witness data is going public, and I am doing it on purpose."*

Without it, the compiler rejects your program:

```compact
witness getSecret(): Bytes<32>;
export ledger stored: Bytes<32>;

export circuit record(): [] {
    stored = getSecret();  // compiler error: undeclared disclosure
}
```

With it:

```compact
export circuit record(): [] {
    stored = disclose(getSecret());  // compiles fine
}
```

The disclosure tracking follows witness data through arithmetic, type conversions, and function calls. You cannot obfuscate your way past it — the compiler's abstract interpreter tracks taint through every operation.

**Best practice:** put `disclose()` as close to the disclosure point as possible, not deep in a helper function.

## When to Use `export ledger`

Use `export ledger` when the data **must be publicly visible for other participants to interact with the contract correctly**.

Good candidates:
- Global invariants (total supply, reserve balance, curve slope)
- State flags others need to react to
- Commitments to private values (the hash, not the value itself)
- Anything your frontend needs to read directly

Bad candidates:
- Per-user balances or allowances
- Personal data of any kind
- Any value that belongs to one user and no one else needs to verify directly

The quick heuristic: if removing this field from the chain would break another user's ability to interact with the contract, it belongs in `export ledger`. If it belongs only to one user, it belongs in private state.

## The Commitment Pattern

If `export ledger` puts values on-chain as plaintext, and private state keeps them off-chain entirely — how does the contract *verify* anything about private state without seeing it?

The answer is commitments. Hash arbitrary data together with a random nonce. Store the hash on-chain. The value stays local.

```compact
// on-chain: store a commitment, not the balance
export ledger balanceCommitments: Map<Bytes<32>, Bytes<32>>;
```

The `CompactStandardLibrary` provides these tools:

```compact
circuit transientHash<T>(value: T): Field;
circuit transientCommit<T>(value: T, rand: Field): Field;
circuit persistentHash<T>(value: T): Bytes<32>;
circuit persistentCommit<T>(value: T, rand: Bytes<32>): Bytes<32>;
```

- `transient*` — use only when the value is **not** kept in ledger state
- `persistent*` — suitable for storage in a contract's `ledger` state

**Critical:** the nonce must never be reused. Two commitments with the same nonce and value are identical on-chain, allowing an observer to link them and breaking the privacy guarantee.

One subtlety: `transientCommit(e)` is treated as non-witness data even if `e` contains witness data (because the nonce hides the input). `transientHash(e)` is not — the hash output is still tracked as witness-tainted.

## Ledger-State Types

A ledger field's type must be one of the following ledger-state types:

| Type | Description |
|------|-------------|
| `T` (any Compact type) | A single `Cell<T>` — readable and writable |
| `Counter` | An unsigned integer counter with `increment`/`decrement` |
| `Set<T>` | An unbounded set of values |
| `Map<K, V>` | An unbounded key-value mapping |
| `List<T>` | An unbounded ordered list |
| `MerkleTree<n, T>` | A bounded Merkle tree of depth `n` (2 ≤ n ≤ 32) |
| `HistoricMerkleTree<n, T>` | Like `MerkleTree` but retains past roots for membership proofs |
| `Kernel` | Special type for built-in ledger operations (block time, balances, minting) |

---

## Cell (Plain Types)

When you declare a ledger field with an ordinary Compact type, it is implicitly a `Cell`:

```compact
ledger F: Uint<16>;

export circuit putF(x: Uint<16>): [] {
  F = disclose(x);       // shorthand for F.write(disclose(x))
}

export circuit getF(): Uint<16> {
  return F;              // shorthand for F.read()
}
```

---

## Counter

A `Counter` is better than a `Cell<Uint<64>>` for values that are only incremented or decremented, because it does not read the current value when updating — reducing on-chain state dependency and lowering the chance of a transaction being rejected.

```compact
import CompactStandardLibrary;
ledger cnt: Counter;

export circuit inc(): [] { cnt += 1; }
export circuit dec(): [] { cnt -= 1; }
export circuit get(): Uint<64> { return cnt; }
```

## Map

```compact
import CompactStandardLibrary;
ledger balances: Map<Bytes<32>, Uint<64>>;

export circuit setBalance(addr: Bytes<32>, amount: Uint<64>): [] {
  balances.insert(disclose(addr), disclose(amount));
}

export circuit getBalance(addr: Bytes<32>): Boolean {
  return balances.member(addr);
}
```

`Map` values can themselves be ledger-state types (except `Kernel`), enabling nested state like `Map<Boolean, Map<Field, Counter>>`.

---

## Set

```compact
import CompactStandardLibrary;
ledger allowlist: Set<Bytes<32>>;

export circuit add(addr: Bytes<32>): [] {
  allowlist.insert(disclose(addr));
}

export circuit isAllowed(addr: Bytes<32>): Boolean {
  return allowlist.member(addr);
}
```

## MerkleTree vs HistoricMerkleTree

Use `MerkleTree` when you only need to prove membership against the **current** root.

Use `HistoricMerkleTree` when you need to prove membership against **past** roots — for example, in nullifier-based systems where a user proves they were in a set at a previous point in time.

```compact
import CompactStandardLibrary;
ledger members: MerkleTree<8, Bytes<32>>;

export circuit addMember(leaf: Bytes<32>): [] {
  members.insert(disclose(leaf));
}
```

`HistoricMerkleTree.checkRoot(rt)` returns `true` if `rt` is any past root, not just the current one.

## The Decision Table

| If your data... | Use... |
|----------------|--------|
| Defines market price or global invariants | `export ledger` |
| Needs to be read by your frontend directly | `export ledger` |
| Is per-user and sensitive | Private state + commitment in `export ledger` |
| Is a temporary computation input | Witness only (no ledger) |
| Proves membership without revealing value | `persistentCommit` in `export ledger` |
| Must never change after deployment | `sealed ledger` |
