# Witnesses

A witness is a callback function that the TypeScript DApp provides to a Compact contract at runtime. Witnesses are how circuits access private data — data that lives on the user's local machine and never touches the chain.

> Source: [docs.midnight.network/compact/reference/compact-reference#declaring-witnesses-for-private-state-management](https://docs.midnight.network/compact/reference/compact-reference#declaring-witnesses-for-private-state-management), [docs.midnight.network/compact/reference/explicit-disclosure](https://docs.midnight.network/compact/reference/explicit-disclosure)

---

## The Mental Model

The name *witness* comes from zero-knowledge literature. In a ZK proof, a witness is the evidence you need to believe a statement is true. In Compact, it is the evidence the contract needs to believe an operation is permissible — for example, the secret key proving you are the one authorized to clear a value.

Witnesses are the bridge between the ZK circuit world (Compact) and the private local world (TypeScript). They supply private inputs to circuits without those inputs ever appearing on-chain.

---

## Declaring a Witness

A witness declaration looks like a circuit declaration but has no body. The body is provided by the TypeScript DApp:

```compact
witness secretKey(): Bytes<32>;
```

```compact
witness getBalance(addr: Bytes<32>): Uint<64>;
```

```compact
witness userNonce(): Field;
```

Witness declarations can appear anywhere among the program elements at the top level or inside a module. They can also be generic:

```compact
witness getItem<T>(index: Uint<32>): T;
```

---

## Calling a Witness

Witnesses are called from inside circuits just like regular functions:

```compact
witness secretKey(): Bytes<32>;

export circuit clear(): [] {
  const sk = secretKey();
  const pk = publicKey(round, sk);
  assert(authority == pk, "Not authorized");
  state = State.UNSET;
  round.increment(1);
}
```

The witness call happens locally on the user's machine. The ZK proof is then generated proving the circuit executed correctly — without revealing `sk` to anyone.

---

## Witness Data is Tracked

The compiler tracks witness data through every operation — arithmetic, type conversions, struct construction, function calls. You cannot hide it. This is called the **witness protection program**.

Any value derived from a witness call is also considered witness data:

```compact
witness getSecret(): Field;

export circuit example(): [] {
  const s = getSecret();
  const doubled = s + s;       // still witness data
  const converted = s as Uint<64>; // still witness data
  ledger = doubled;            // compiler error: undeclared disclosure
}
```

---

## Witness Data and `disclose()`

When witness data needs to flow into the public ledger, into a return value of an exported circuit, or into another contract — you must explicitly wrap it in `disclose()`.

**Without `disclose()` — compiler error:**

```compact
witness getBalance(): Bytes<32>;
export ledger balance: Bytes<32>;

export circuit record(): [] {
  balance = getBalance();  // error: potential witness-value disclosure must be declared
}
```

**With `disclose()` — compiles:**

```compact
export circuit record(): [] {
  balance = disclose(getBalance());
}
```

`disclose()` does not encrypt or hide the value. It is a compile-time annotation that says: *"I know this witness data is going public, and I am doing it intentionally."*

---

## The Error Message

When you forget `disclose()`, the compiler tells you exactly where the witness data came from, what path it took, and where the disclosure is happening:

```
Exception: line 6 char 11:
  potential witness-value disclosure must be declared but is not:
    witness value potentially disclosed:
      the return value of witness getBalance at line 2 char 1
    nature of the disclosure:
      ledger operation might disclose the witness value
    via this path through the program:
      the right-hand side of = at line 6 char 11
```

Read the path from top to bottom — it traces exactly how the witness data traveled to the disclosure point.

---

## Indirect Disclosure is Also Caught

The compiler detects disclosure even when witness data travels through arithmetic, structs, and function calls:

```compact
circuit obfuscate(x: Field): Field {
  return x + 73;  // still witness data
}

export circuit record(): [] {
  const s = S { x: getBalance() as Field };
  const x = obfuscate(s.x);
  balance = x as Bytes<32>;  // compiler still catches this
}
```

Even returning a comparison result from an exported circuit counts as disclosure:

```compact
witness getBalance(): Uint<64>;

export circuit balanceExceeds(n: Uint<64>): Boolean {
  return getBalance() > n;  // compiler error: comparison result discloses witness data
}
```

---

## Where to Place `disclose()`

Best practice: place `disclose()` **as close to the disclosure point as possible**.

For structured values (tuples, vectors, structs), wrap only the portions that contain witness data — not the whole structure — to avoid accidentally disclosing non-private fields.

For a witness that always returns data that is already safe to disclose, you can wrap the call itself:

```compact
export circuit record(): [] {
  balance = disclose(getBalance());  // wrap the call
}
```

---

## Safe Standard Library Routines

The compiler knows that certain standard library operations sufficiently hide witness data:

- `transientCommit(e)` — treated as **non-witness data** even if `e` contains witness data. The random nonce provides sufficient hiding.
- `transientHash(e)` — still treated as **witness data**. A bare hash may not sufficiently hide the input.

This means `transientCommit` can be used to commit private values for on-chain storage without triggering the disclosure requirement on the commitment itself — only the value you're committing remains tracked.

---

## Critical: Witness Results Are Untrusted

> Do not assume in your contract that the code of any `witness` function is the code that you wrote in your own implementation. Any DApp may provide any implementation it wants for your `witness` functions. Results from them should be treated as untrusted input.

This is a fundamental design point. The ZK proof guarantees that the circuit's *logic* ran correctly given whatever inputs the witnesses returned — it does not guarantee that the witnesses returned sensible values. Your contract must verify witness outputs with `assert` where correctness matters.

---

## Witness vs Circuit — Key Differences

| | `witness` | `circuit` |
|--|-----------|-----------|
| Body | Implemented in TypeScript | Implemented in Compact |
| Execution | Runs locally on user's machine | Compiled to ZK circuit |
| Output | Private data (witness data) | Proven result |
| Trust | Untrusted — must validate | Enforced correct by ZK proof |
| Can access private state | Yes | Only via witness calls |

## Full Example

```compact
pragma language_version 0.22;
import CompactStandardLibrary;

export ledger authority: Bytes<32>;
export ledger round: Counter;

witness secretKey(): Bytes<32>;

circuit publicKey(r: Field, sk: Bytes<32>): Bytes<32> {
  return persistentHash<Vector<3, Bytes<32>>>(
    [pad(32, "midnight:lock:pk"), r as Bytes<32>, sk]
  );
}

export circuit clear(): [] {
  const sk = secretKey();           // call witness: private input
  const pk = publicKey(round, sk);  // compute: still private
  assert(authority == pk, "Not authorized");  // verify: enforced by ZK
  round.increment(1);
}
```

In `clear`, `sk` is never disclosed. The ZK proof proves that whoever called `clear` knew a secret key that hashes to `authority` — without revealing the key itself.
